IPv6 Migration Performance Dashboard

The exhaustion of IPv4 addresses is forcing organizations worldwide to migrate to IPv6. However, there is no single migration path — different strategies suit different environments. This research evaluates three IPv6 transition mechanisms to determine which provides the best balance of performance, reliability, and compatibility.

Research Problem
Problem 1
IPv6 Migration and Specific Environments
Not every migration strategy works for every organization. ISPs, enterprises, and mobile networks have fundamentally different infrastructure needs. Choosing the wrong strategy leads to poor performance, compatibility issues, and wasted resources.
Problem 2
Balancing Network Performance with IPv4 Address Conservation
The ideal solution must preserve IPv4 connectivity while maximizing performance on IPv6 infrastructure. Strategies that save addresses often sacrifice speed. Strategies that prioritize speed often waste addresses. Finding the right balance is critical.
Problem 3
The Hassle of Data Collection
Meaningful performance comparison requires consistent, repeatable testing across multiple strategies — a process that is time-consuming, error-prone, and complex. Without reliable data, migration decisions become guesswork.

Our Approach: We built a controlled GNS3 network lab to test all three strategies under identical conditions. Every test was automated, repeated multiple times, and results are visualized in real-time through this dashboard. This provides objective, data-driven evidence to guide IPv6 migration decisions.

The Three Strategies Under Evaluation
Dual Stack
Runs both IPv4 and IPv6 simultaneously. Devices communicate using whichever protocol they prefer. No translation or tunneling required.
Best for: Enterprises with existing IPv4 infrastructure transitioning gradually
DS-Lite (Dual-Stack Lite)
Sends IPv4 traffic through an IPv6 tunnel. The core network runs IPv6 while preserving IPv4 connectivity for end users.
Best for: ISPs with IPv6 backbone needing to support legacy IPv4 customers
NAT64 / DNS64
Translates between IPv6 and IPv4 at the network edge. IPv6-only clients can reach IPv4-only servers through address translation.
Best for: Mobile networks and IPv6-only deployments needing IPv4 access
How We Tested: 4 Performance Dimensions
Test 1: Packet Size Performance
TCP Throughput • UDP Jitter • ICMP Latency • Packet Loss
Tests 5 data sizes (5M to 20M) to measure scalability. Reveals if performance degrades under increasing network load.
Test 2: HTTP Download Speed
Download Speed (1KB to 1MB) • Drop Detection • MSS Clamping
Simulates real-world web browsing. Detects critical speed drops that would impact user experience.
Test 3: RTT Latency Consistency
Min/Avg/Max RTT • Jitter (Mdev) • Packet Loss (3 runs × 100 pings)
Repeated 3 times to verify consistency. Low latency = better video calls, gaming, and responsive applications.
Test 4: Fragmentation Boundary
MTU Discovery • Fragmentation Success Rate • RTT Overhead
Finds the maximum packet size before splitting occurs. Critical for VPN performance, streaming quality, and network efficiency.
